## fnet1 LVM Rebuild Narrative

### Context
Rebuilding fnet1 storage from manual 3-disk partitions (sda, sdb, sdc) to a single LVM pool. The naive approach — unmount → wipe → pvcreate — failed in unexpected ways.

### What Went Wrong
The `wipefs -a /dev/sda` command returned **"Device or resource busy"** despite no visible processes holding the device.

### Diagnosis Steps (What Worked vs. What Failed)

| Step | Tool | Result | Why |
|------|------|--------|-----|
| 1 | `lsof /dev/sda` | Nothing | Block device holders don't always show in lsof |
| 2 | `fuser /dev/sda` | Nothing | Same — doesn't catch mount namespaces |
| 3 | `lslocks` | Nothing | File locks, not mount namespaces |
| 4 | `wipefs -a` | "Device or resource busy" | GPT backup headers pinned by namespace refs |
| 5 | `partx -d /dev/sda` | "error deleting partitions" | Kernel holds active references |
| 6 | `blockdev --rereadpt /dev/sda` | "ioctl error BLKRRPART" | Device busy at block level |
| ✅ 7 | Search `/proc/*/mountinfo` for `/var` or `/tmp` | **Found it** | systemd-resolved (PID 1033) had private bind mount: `372 348 8:33 / /var ro,nosuid,relatime shared:182 - ext4 /dev/sdc1` |

### Root Cause
Systemd services with `ProtectSystem` and `PrivateTmp` directives create **private mount namespaces** that hold block devices open via bind mounts. These are invisible to standard process and lock inspection tools. The services include:

- `systemd-resolved`
- `systemd-journald`
- `rsyslog`
- `polkitd`
- `cron`
- `dbus-daemon`

All have `ProtectSystem` directives in their service files.

### Why Lazy Unmount Failed
`umount -l /var` detaches the filesystem from the mount tree but the kernel still holds the block device reference because systemd services inside their namespaces still have active bind mounts. The device appears "busy" at the block level even though no process list shows a holder.

### The Pivot
**Live LVM migration on a systemd-protected system is unreliable.** After multiple failed attempts to force unmounts, the agent pivoted to a **one-shot early-boot systemd service** approach:

1. Create a systemd service that runs after `local-fs.target` (rootfs + staging mounted)
2. Runs before `sysinit.target` (before any `ProtectSystem`/`PrivateTmp` services start)
3. The service script: stop services → unmount → wipe → pvcreate → vgcreate → lvcreate → mkfs → mount → restore data → update fstab → reboot

This avoids the mount namespace problem entirely because no services are running when the script executes.

### Key Learnings
1. **Never trust `lsof` alone for block device conflicts** — always check `/proc/*/mountinfo` for mount namespaces
2. **Systemd `ProtectSystem` is a hidden mount holder** — services you didn't start can pin your disks
3. **Live LVM migration on systemd-protected systems** requires early-boot service or initramfs hook
4. **Staging data before destructive ops is essential** — we staged ~34GB to NVMe before hitting the partition lock issue
5. **When standard tools cascade-fail, check for mount namespaces before escalating to reboot**
