### Background Watcher: `scripts/decompose-watcher.py`

**New file: `decompose-watcher.py`**
- Polls the pending trigger directory for new triggers
- For each trigger:
  1. Generates a PLAN markdown from a template with generic sub-tasks
  2. Calls `decompose_task.py --plan <plan> [--submit|--dry-run]`
  3. Monitors node completion (5s wait, then collects)
  4. Calls `synthesize_results.py --plan <id> --json`
  5. Writes result to `.pi/decompose-triggers/results/`
  6. Moves trigger to completed directory

**Usage:**
```bash
# Poll once (dry-run)
python3 technical-infrastructure/scripts/decompose-watcher.py --poll

# Poll once and actually execute
python3 technical-infrastructure/scripts/decompose-watcher.py --poll --execute

# Continuous watch loop
python3 technical-infrastructure/scripts/decompose-watcher.py --watch --execute

# List pending/completed
python3 technical-infrastructure/scripts/decompose-watcher.py --list-pending
python3 technical-infrastructure/scripts/decompose-watcher.py --list-completed
```

### Step 1: Enable auto-decomposition

Add to your `.pi/keyword-router.json`:
```json
{
  "enableAutoDecomposition": true,
  "decompositionNodePool": ["fnet2", "fnet3", "fnet4", "fnet5", "fnet6"],
  "maxDecompositionSteps": 5
}
```

In pi, run: `/keyword-route-decompose`

Shows pending and completed triggers with their status.

## Known Limitations

1. **Watcher does not truly monitor nodes** — it waits 5s then collects. A production implementation would poll node task directories until all tasks complete.
2. **Auto-generated PLANs are generic** — they use fixed sub-task templates. A production implementation would use the LLM to generate task-specific sub-tasks.
3. **Result injection not wired** — synthesized results are written to files but not automatically injected into the next prompt context. The user must manually read the result file.
4. **No error recovery** — if a node fails mid-task, the watcher reports failure but does not retry or escalate.

These are acceptable for Phase 2A MVP. Phase 2B (adaptive feedback) and future iterations can address them.
