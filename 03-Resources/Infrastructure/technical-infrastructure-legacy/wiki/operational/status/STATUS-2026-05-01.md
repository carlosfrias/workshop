## UPDATE: fnet1 LVM Rebuild Complete (2026-05-01 10:15)

**Status:** Complete. fnet1 now has clean LVM across 3√ó931GB disks.

### Resolution Summary
- **Root cause of initial failure:** Mount order bug in early-boot script ‚Äî tried to mount `/var/log` before `/var` was mounted, causing `mount: mount point does not exist`
- **Fix:** Updated script to mount parents first, then mkdir children inside mounted parents, then mount children
- **Also fixed:** Updated cp -a to rsync -a for data restore (handles symlinks correctly)
- **Also fixed:** Added sed cleanup for old fstab entries before building new ones
- **Also fixed:** Decomposition documentation updated with corrected mount order
- **Also discovered:** Ubuntu unattended-upgrade hooks shutdown sequence, delaying reboot. Future early-boot services should mask unattended-upgrades first.

### Why This Took So Long
1. Initial attempt: live unmount failed (systemd private mounts)
2. Pivot to early-boot service approach
3. Service failed due to mount order bug
4. Reboot stuck in unattended-upgrade for ~15 min
5. Post-reboot: second boot auto-mounted LVs via unknown mechanism
6. Manual fix: data restore, fstab rebuild, service cleanup

### Key Lesson
Early-boot service script ordering is critical when parent/child mount relationships exist (e.g., /var/log inside /var). Mount parent -> mkdir children -> mount children.

- decomposition-examples/systemd-mount-lock/05-prompt-pivot-to-early-boot.md ‚Äî corrected mount order
- decomposition-examples/systemd-mount-lock/scripts/fnet1-lvm-setup.sh ‚Äî corrected mount order
