### Thinking — What the Problem Actually Is

Given the above, the decomposition needed is NOT about fitting within context windows. It's about **structural decomposition for local model effectiveness**:

1. **AGENTS.md files are mixed-use** — they contain routing rules, quality checklists, planning gates, conventions, cross-domain references, and operational procedures. A single inference over the whole file means the model must simultaneously classify the user's intent AND cross-reference against conventions. This is hard for 4B–8B models.

2. **The files have clear structural boundaries** — sections like "Domain Activation Check", "Model Routing", "Quality Checklist", "Rules", "Planning Quality Gates" could each be separate files or indexed chunks.

3. **Different sections serve different cognitive loads**:
   - Domain activation → pattern matching (lightweight, could be keyword-based)
   - Quality checklist → procedural verification (structured, could be a form)
   - Planning gates → reasoning + synthesis (needs heavier model)
   - Cross-domain references → lookup (could be indexed)

4. **The decomposition examples** (`systemd-mount-lock` = 10 small prompts, `network-troubleshooting` = 34 small prompts) show success with SMALL, FOCUSED prompts to local models. The principle is: give the model ONE job per prompt, not a whole instruction manual.

### Hypothesis — Root Cause
The real issue is that `AGENTS.md` and `AGENTS-full.md` are **monolithic instruction sets**. A local model receiving the full file must:
1. Parse hundreds of rules
2. Map user prompt to domain
3. Check quality gates
4. Verify compliance

All in one pass. This is equivalent to asking a junior engineer to simultaneously read the employee handbook, write code, and review a PR — while the handbook is on their lap but they can't bookmark pages.

### Hypothesis — Solution Direction
Extract AGENTS.md into **queryable, indexed modules** that are loaded on demand based on the task phase:

| Phase | Module to Load | Why |
|-------|---------------|-----|
| Domain activation | `agents/domain-routing.json` (or small .md) | Only keywords + file paths |
| Model routing | `agents/model-routing.json` | Only route table, no prose |
| Quality checklist | `agents/quality-gates.md` | Loaded AFTER domain is known |
| Planning gates | `agents/planning-gates.md` | Only if creating a PLAN |
| Cross-domain refs | `agents/cross-domain-refs.md` | Only if task is multi-domain |

This mirrors the existing "prompt-triggered references" pattern in `ti/AGENTS.md` but makes it **first-class and machine-queryable** rather than manual keyword matching.

## Raw Thinking — Why This Approach Is Right

The user asked specifically about **file ingestion and parsing** by local models. After audit, the files are NOT too large for context windows. But they ARE too *structurally dense*. A small model given a 400-line instruction manual with 15 different rule types cannot reliably activate the right section at the right phase.

The existing decomposition pipeline (PLAN-2026-05-02-1930) solves **task decomposition** — breaking a user request into sub-tasks. What is missing is **system prompt decomposition** — breaking the AGENTS instructions themselves into load-on-demand modules.

If AGENTS.md becomes a router ("Which phase are you in? → Load module X"), then local models only ever operate on focused, bounded instruction sets. This is exactly what the decomposition examples do for user prompts (10 small prompts instead of 1 big one), but applied to the system itself.

This is self-referential improvement: the decomposition framework should decompose its own instructions.
