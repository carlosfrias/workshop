# Planning: Meta-Orchestration Framework (Local-First LLM Routing + Decomposition)
**Date:** 2026-05-01 16:45  
**Status:** Architectural proposal — needs prioritization and decomposition  
**Context:** Cloud model usage has dominated recent sessions despite 7 local nodes being available. The infrastructure exists; the integration layer does not.

---

## Problem Statement

**What we observe:**
- kimi-k2.6 (1042B cloud, ultra-reasoning) is handling prompts that qwen3:8b or gemma4:e4b could manage
- Routing is keyword-based, not complexity-based — "think deeply" triggers ultra-reasoning even for simple analysis
- No decomposition is happening at runtime — complex prompts go to single expensive model instead of being fanned out to multiple cheap models
- No performance feedback loop — we don't know which prompt types succeed on which local models
- Cloud cost accumulates while local compute sits idle

**What we want:**
- Every prompt evaluated for complexity before routing
- Simple prompts → local model (qwen3.5:4b, ~0.1s response, $0 cost)
- Medium prompts → local model (qwen3:8b or gemma4:e4b, ~2-5s, $0 cost)
- Hard prompts → decompose into sub-tasks → fan out to multiple local nodes → synthesize
- Only when local decomposition fails or sub-tasks time out → escalate specific sub-task to cloud
- Monitor and learn: which prompt types need which model tier

## Key Design Decisions

### 1. Classification Must Be Fast and Cheap

The classifier itself must run on qwen3.5:4b (fastest, smallest) with a simple prompt:

```
Classify this task complexity: TRIVIAL, SIMPLE, MEDIUM, HARD

Examples:
- "Check if fnet2 is online" → TRIVIAL (single SSH ping)
- "Summarize this 3-paragraph email" → SIMPLE (single model call)
- "Design an Ansible playbook for vault deployment" → MEDIUM (structured output, needs planning)
- "Analyze market regime shifts across 5 timeframes with backtesting" → HARD (multi-step, uncertain decomposition)

Task: {user_prompt}
Complexity:
```

If classification itself takes >0.5s or fails, default to cloud (safe but expensive). Track this.

### 2. Decomposition Uses Existing PLAN Format

When a HARD prompt arrives, the system should:
1. Look for existing PLAN templates that match the task type (e.g., "Ansible Vault Deployment" → `PLAN-2026-05-01-0915.md`)
2. Use the PLAN's structure to generate sub-task JSONs
3. Each sub-task gets a model assignment:
   - "Write playbook skeleton" → qwen3:8b (structured generation)
   - "Validate syntax" → qwen3.5:4b (fast check)
   - "Generate documentation" → qwen3:8b (prose)
   - "Test on one node" → shell command via task-worker.sh

### 3. SESSION-NOTES Feed the Knowledge Base (Machine-Readable)

After each session, the system should parse structured tracking tables from SESSION-NOTES:

**From the Model Performance Log:**
| Prompt Type | Model Used | Adequate? | Cost |
|-------------|-----------|-----------|------|
| Hardware spec extraction | gemma4:e4b | Yes | $0 |
| LVM diagnosis reasoning | kimi-k2.6 | Yes | $0.05 |

**From the Pattern Analysis:**
| Finding | Implication |
|---------|-------------|
| Markdown formatting trivial for qwen3.5:4b | Never route formatting to cloud |
| systemd + LVM cross-domain needs cloud | Keep ultra-reasoning for >2-domain integration |

The system uses these tables to:
- **Update classifier few-shot examples** — if a SIMPLE prompt was misclassified as HARD, add it as a counter-example
- **Update decomposition templates** — if an agent found a new approach, add it to the PLAN template library
- **Update routing rules** — if gemma4:e4b consistently succeeds on "extraction" prompts, hard-route all extraction to gemma4

**This is the critical feedback loop.** The documentation is not just for humans — it's training data for the adaptive system.

**Template format:** See [`wiki/templates/SESSION-NOTES-template.md`](../../templates/SESSION-NOTES-template.md) for the machine-readable tracking table format.

### 4. PLAN Documents Are Executable Specifications

PLAN documents already contain decomposition steps. With the addition of the **Decomposition Tracking Table** and **Execution Results Table**, a PLAN becomes:
1. A human-readable plan (what we already have)
2. A machine-parseable execution specification (what to run, on which model/node)
3. A post-hoc performance report (what actually happened)

This means the system can:
- Read an old PLAN for a similar task
- Extract the decomposition steps and model assignments
- Execute them via the orchestration system
- Compare actual vs. planned performance
- Update the PLAN with new learnings

**Template format:** See [`wiki/templates/PLAN-template.md`](../../templates/PLAN-template.md) for the decomposition tracking format.

### 5. Node-Model Matching

From `lab-capacity-report.json`:

| Node | Models | Best For |
|------|--------|----------|
| fnet2 | qwen3.5:4b, qwen3:8b | Fast tasks, small context |
| fnet3 | All 3 (depot) | Any task, closest to orchestrator |
| fnet4-fnet6 | All 3 | Medium tasks, parallel execution |
| fnet7 | qwen3.5:4b, qwen3:8b | Fast tasks, fallback |

Task assignment should consider:
- Model availability (does node have the required model loaded?)
- Current load (is node already processing?)
- Network proximity (fnet3 is physically close, lower latency)
- Context size (larger models need more RAM — check node capacity)

### 5. Escalation Is Per Sub-Task, Not Whole Prompt

If a HARD prompt decomposes into 5 sub-tasks and 4 succeed on local models but 1 fails (e.g., "generate complex regex pattern"), only that 1 sub-task goes to cloud. The other 4 stay local. This maximizes cost savings.

## Phased Implementation

### Phase 1: Classification + Simple Routing (Week 1)
**Goal:** Stop sending SIMPLE prompts to cloud models.

- Build `classify_prompt.py` — runs qwen3.5:4b locally, returns TRIVIAL/SIMPLE/MEDIUM/HARD
- Extend `pi-keyword-router` with complexity routing layer
- TRIVIAL/SIMPLE → local model (qwen3:8b or gemma4:e4b)
- MEDIUM/HARD → current behavior (cloud or local depending on keywords)
- **Impact:** Immediate 40-60% reduction in cloud calls for routine tasks

### Phase 2: Decomposition Engine (Week 1-2)
**Goal:** MEDIUM prompts get decomposed and distributed.

- Build `decompose_task.py` — reads PLAN templates, generates sub-task JSONs
- Integrate with `submit_task.py` — fan out to nodes
- Build `synthesize_results.py` — combine sub-outputs
- **Impact:** MEDIUM prompts that previously went to cloud now stay local

### Phase 3: Performance Monitor (Week 2)
**Goal:** Know what's working and what isn't.

- Log every prompt: type, model, latency, tokens, cost, success/failure
- Weekly report: "These 5 prompt types always fail on local models"
- Dashboard or simple markdown report
- **Impact:** Data-driven routing decisions

### Phase 4: Adaptive + Feedback (Week 3)
**Goal:** System improves itself and justifies its cost structure.

- **Auto-update classifier** few-shot examples from performance logs
- **Auto-update PLAN templates** from successful decompositions
- **Auto-generate SESSION-NOTES summaries** for knowledge base
- **Tier adjustment engine:**
  - If Cloud Standard succeeds 90%+ on a prompt type previously routed to Cloud Premium → downgrade tier
  - If Local Standard succeeds 90%+ on a prompt type previously routed to Cloud Standard → downgrade tier
  - If local model failure rate increases for a prompt type → upgrade tier (with evidence)
- **Cost justification report:** Monthly summary showing
  - Total cloud spend
  - % justified (local model genuinely failed)
  - % avoidable (should have been local)
  - Trend over time

**Impact:** Long-term cost reduction accelerates; remaining cloud spend is fully justified with evidence.

### Phase 5: Documentation-to-System Feedback (Continuous)
**Goal:** Documentation becomes the system's institutional memory.

- Parse new PLAN/SESSION-NOTES documents as they're created
- Extract decomposition patterns that worked → add to template library
- Extract model performance patterns → update routing rules
- Extract unjustified cloud usage → flag for routing rule tightening
- Generate weekly "routing effectiveness report" from documentation

## Cost Model (Tiered with Justification)

| Tier | Model | Cost/Prompt | When Justified |
|------|-------|-------------|----------------|
| **Local Fast** | qwen3.5:4b | $0 | Trivial tasks: <500 tokens, single domain, deterministic |
| **Local Standard** | qwen3:8b / gemma4:e4b | $0 | Standard tasks: 1-2 domains, <2000 tokens, structured output |
| **Cloud Standard** | qwen3.5:397b | ~$0.005 | Complex tasks: 3+ domains, reasoning, planning, novel but bounded |
| **Cloud Premium** | kimi-k2.6 | ~$0.01-0.05 | Cross-domain integration, creative synthesis, failure mode analysis |

**Critical Principle:** Cloud Premium is not the default for "hard." It is reserved for prompts where the performance log shows:
1. Local models *consistently fail* on this prompt type, OR
2. The quality gap between Cloud Standard and Cloud Premium is *material* to the outcome

**Justification Evidence:**
- For Cloud Standard: "Local model X failed on 3/3 attempts for this prompt type"
- For Cloud Premium: "Cloud Standard produced output that was inadequate — required cross-domain synthesis that only ultra-reasoning model handles"

| Scenario | Current Cost | With Framework | Savings | Notes |
|----------|-------------|----------------|---------|-------|
| 100 SIMPLE prompts/day | $3-5 (cloud default) | $0 (local) | 100% | All local |
| 50 MEDIUM prompts/day | $2-4 (cloud default) | $0.25 (mostly local + selective cloud) | 85-95% | Decomposition keeps 90% local |
| 20 HARD prompts/day | $2-3 (all cloud) | $0.50-1.00 (selective cloud) | 50-75% | Only sub-tasks that genuinely need cloud |
| **Daily total** | **$7-12** | **$0.75-1.25** | **75-90%** | **With full justification trail** |

**Monthly projection:** $210-360/month → $22-37/month

**But more importantly:** The remaining $22-37/month is *justified* with evidence. Every cloud prompt has a documented reason why local models couldn't handle it.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Local models hallucinate on complex tasks | High | Wrong answers | Always verify output before acting; fallback to cloud on uncertainty |
| Classification mis-routes HARD as SIMPLE | Medium | Local model fails silently | Monitor and alert on failure patterns; classifier improves over time |
| Orchestration adds too much latency | Medium | User experience degrades | Set timeout thresholds; if total local latency > cloud latency, route to cloud |
| Node failures mid-task | Low | Lost work | Nodes are stable; retry logic in task-worker.sh handles failures |
| Context too large for local models | Medium | OOM or truncation | Pre-filter context size; route large context to cloud automatically |
