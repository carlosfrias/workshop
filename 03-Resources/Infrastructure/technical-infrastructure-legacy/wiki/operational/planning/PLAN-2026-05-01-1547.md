# Planning: Local Network Orchestration for Lab Nodes
**Date:** 2026-05-01 15:47  
**Context:** The lab has 7 compute nodes (fnet1-fnet7) plus this orchestrator. Currently work distribution is ad hoc — Ansible playbooks for homogeneous tasks, SSH commands for one-offs, background agents limited to one task at a time.

## Problem Statement

**What we lack:** A systematic way to distribute heterogeneous work across multiple nodes, track progress, collect results, and handle failures — all over the local network without depending on external services.

**Current state:**
- Ansible: excellent for "do the same thing on all nodes" (playbooks)
- SSH: good for one-off commands, terrible for state tracking
- Background agents: good for single-node long tasks, no cross-node coordination
- Gist protocol: async, works through firewalls, but **worker side doesn't self-poll** — requires manual prompting every cycle

**What we need:**
- Task queue visible to orchestrator
- Worker nodes claim tasks autonomously
- Result stream back to orchestrator
- Failure detection and retry

## Proposed Solutions

### Option A: pi-intercom (Primary)
**What it is:** pi skill for peer-to-peer messaging between pi sessions on the same machine or LAN.

**Strengths:**
- Already exists (`pi install pi-intercom`)
- Designed for exactly this: session-to-session messaging
- No external dependencies (LAN only)
- Real-time (not polling-based)

**Challenges:**
- Requires pi running on each lab node (not currently deployed)
- Message format is unstructured — we'd need a protocol layer on top
- No built-in task queue or retry logic

**Work needed:**
1. Deploy pi + pi-intercom to fnet1-fnet7 (Ansible playbook)
2. Define message protocol: `task_id`, `node`, `type` (claim/result/error), `payload`
3. Build task queue on orchestrator (filesystem or simple JSON)
4. Build worker loop on nodes (check intercom → execute → report)
5. Build orchestrator UI (just `intercom list` + `intercom send`)

### Option B: SSH + JSON files (Simpler, No New Stack)
**What it is:** Orchestrator writes task JSON files to `/srv/tasks/` via SSH/SCP. Workers poll directory via cron or systemd timer, execute, write result JSON.

**Strengths:**
- Uses only SSH (already working)
- No new software on nodes
- Filesystem is natural queue
- Easy to inspect state (`ls /srv/tasks/`)

**Challenges:**
- Polling latency (cron every minute)
- Requires shared filesystem or orchestrator as file server
- Race conditions without locking

**Work needed:**
1. Create `/srv/tasks/{pending,running,completed}` on fnet1 (or orchestrator)
2. Mount shared tasks dir on all nodes via NFS
3. Build task generator (creates JSON in `pending/`)
4. Build worker service (systemd timer or cron job that polls `pending/`)
5. Build task executor (reads JSON, runs command, writes result to `completed/`)

### Option C: Gist Protocol (Backup, Firewalled Scenarios)
**What it is:** Use existing gist-message-queue skill but fix the worker self-polling problem.

**Strengths:**
- Works through any firewall (only needs HTTPS)
- Persistent audit trail
- Already has partial implementation

**Challenges:**
- Rate limits (gist API)
- Worker must poll continuously
- Higher latency than LAN

**Work needed:**
1. Fix worker to auto-poll without human prompting
2. Add systemd timer for `gist-mq recv` on each node
3. Add timeout and retry logic

## Recommended Approach: Option B (SSH + NFS queue) as MVP, Option A (pi-intercom) as v2

**Rationale:** Option B requires zero new infrastructure. NFS is one `apt install nfs-common` + one `mount` away. The task queue is just files. This can be working in under an hour.

Option A requires deploying Node.js + pi on all nodes. That's a real software distribution problem. Worth doing, but not the fastest path to value.

### MVP Design (1 hour)

**Orchestrator (this node):**
```python
# submit_task.py — runs on orchestrator
import json, uuid, os, subprocess
from datetime import datetime

def submit_task(node, task_type, command):
    task_id = str(uuid.uuid4())[:8]
    task = {
        "id": task_id,
        "node": node,
        "type": task_type,
        "command": command,
        "status": "pending",
        "submitted": datetime.now().isoformat(),
    }
    # Write to orchestrator's task dir
    os.makedirs("tasks/pending", exist_ok=True)
    with open(f"tasks/pending/{task_id}.json", "w") as f:
        json.dump(task, f)
    
    # SCP to node's task queue
    subprocess.run([
        "scp", f"tasks/pending/{task_id}.json",
        f"friasc@{node}:/srv/tasks/pending/"
    ])
    return task_id
```

**Worker (each node):**
```bash
#!/bin/bash
# /usr/local/bin/task-worker.sh
# Run by cron every minute

PENDING="/srv/tasks/pending"
RUNNING="/srv/tasks/running"
COMPLETED="/srv/tasks/completed"

for task in "$PENDING"/*.json; do
    [ -f "$task" ] || continue
    
    # Claim task (move to running)
    id=$(basename "$task" .json)
    mv "$task" "$RUNNING/$id.json"
    
    # Execute
    cmd=$(jq -r .command "$RUNNING/$id.json")
    output=$(bash -c "$cmd" 2>&1)
    rc=$?
    
    # Write result
    jq --arg out "$output" --arg rc "$rc" --arg end "$(date -Iseconds)" \
       '.result = $out | .rc = ($rc | tonumber) | .completed = $end | .status = "completed"' \
       "$RUNNING/$id.json" > "$COMPLETED/$id.json"
    
    # Clean running
    rm "$RUNNING/$id.json"
done
```

**Result collection:**
```bash
# On orchestrator
cd tasks/completed
scp friasc@fnet{i}:/srv/tasks/completed/*.json ./
```
